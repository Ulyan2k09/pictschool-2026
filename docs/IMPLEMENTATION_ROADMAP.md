# Roadmap реализации MVP

Документ задает порядок сборки MVP. Он нужен, чтобы кураторы могли двигаться от минимально работающей системы к финальному сценарию без лишних архитектурных развилок.

## Этап 1. Backend state и event log

Сначала собрать бэкенд как источник правды.

Готово, когда:

- есть состояние миссии `idle`, `running`, `failed`, `completed`;
- есть модель поля, объектов и платформы;
- `POST /api/mission/start` создает стартовое состояние;
- `GET /api/mission` возвращает состояние;
- каждое изменение записывается в журнал событий;
- `GET /api/events` возвращает журнал.

## Этап 2. Simulation / platform adapter

Затем подключить симуляцию или адаптер физической платформы.

Готово, когда:

- бэкенд может отправить команду движения;
- платформа или симуляция возвращает позицию, статус и ошибку;
- невозможное движение не выполняется;
- reset возвращает систему в стартовое состояние;
- есть безопасный режим для тестов.

## Этап 3. Mobile state view и player commands

После появления состояния и движения подключить интерфейс игрока.

Готово, когда:

- приложение показывает поле, платформу, объекты и цель;
- игрок может отправить базовые команды;
- приложение показывает ошибки `path_blocked`, `platform_error`, `unknown_command`;
- игрок видит журнал или краткую ленту событий;
- интерфейс не скрывает действия ИИ.

## Этап 4. AI agent action loop

Добавить ИИ как отдельный компонент, который вмешивается через бэкенд.

Готово, когда:

- ИИ получает снимок состояния миссии;
- ИИ выбирает действие `move_object` или `block_path`;
- `POST /api/ai/action` валидирует действие;
- объект перемещается только через бэкенд;
- в журнале видно, что сделал ИИ и почему.

## Этап 5. Realtime updates

Подключить live-обновления для приложения и агента.

Готово, когда:

- `GET /api/live` отдает события через WebSocket или SSE;
- приложение обновляет поле без ручного refresh;
- события `platform.updated`, `ai.object_moved`, `ai.path_blocked` видны в реальном времени;
- потеря соединения не ломает миссию, потому что состояние можно заново получить через `GET /api/mission`.

## Этап 6. Интеграционные проверки

Собрать все компоненты в повторяемый прогон.

Готово, когда проходят сценарии:

- нормальный проход без вмешательства ИИ;
- ИИ передвинул объект и заблокировал короткий маршрут;
- команда движения отклонена из-за препятствия;
- платформа или симуляция вернула ошибку;
- миссия сброшена и повторно запущена.

## Минимальный порядок для кураторов

1. Зафиксировать стартовую карту и объекты.
2. Реализовать backend state и event log.
3. Подключить симуляцию до физической платформы.
4. Добавить мобильный экран состояния.
5. Добавить команды игрока.
6. Добавить первое действие ИИ `move_object`.
7. Добавить live-обновления.
8. Провести интеграционный прогон.
9. Только после этого усложнять ИИ, задания и финальный баланс.
