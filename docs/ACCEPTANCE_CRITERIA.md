# Критерии готовности MVP

Чеклист актуализирован под механику "agent vs robot, сбор уточек".

## Перед стартом школы

- MVP запускается по инструкции без ручных правок кода.
- `POST /api/round/start` запускает новый раунд.
- `GET /api/round` возвращает состояние раунда.
- `POST /api/turn/submit` принимает ход активного участника.
- `GET /api/events` возвращает журнал событий.
- `POST /api/round/reset` сбрасывает состояние.
- На поле есть оба участника (`agent`, `robot`) и набор уточек.

## Готовность backend

- Backend является единственным источником правды.
- Проверяется очередность ходов (`activeActor`).
- Проверяется лимит команд: максимум 5 за ход.
- Проверяется валидность кодов команд (`1..4`).
- Команда отправляется в симуляцию TCP-строкой по `localhost`.
- Порт команд вынесен в конфиг (`SIM_TCP_COMMAND_PORT=5055`).
- Резервный порт для обратного канала зафиксирован (`SIM_TCP_TELEMETRY_PORT=5056`).

## Готовность simulation

- Симуляция слушает командный TCP-порт на `localhost`.
- Симуляция парсит строку формата `1 2 3 4`.
- Симуляция корректно обрабатывает:
  - движение вперед/назад;
  - повороты влево/вправо на 90 градусов.
- Результат хода возвращает итоговую позицию и собранные уточки.
- Ошибки симуляции транслируются backend как `simulation_error`.

## Готовность клиента

- Клиент показывает текущего активного участника.
- Клиент показывает счет `robot` и `agent`.
- Клиент показывает количество оставшихся уточек.
- Клиент показывает завершение раунда, когда уточек не осталось.
- Ошибки `unknown_command`, `turn_limit_exceeded`, `simulation_error` отображаются явно.

## Перед финалом

- Раунд стабильно завершается событием `round.completed`.
- Журнал событий позволяет восстановить ход игры.
- Очки совпадают с фактическим количеством собранных уточек.
- Очередность ходов не нарушается во всех тестовых сценариях.
