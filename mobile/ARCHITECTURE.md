# Архитектура мобильного приложения

MVP-клиент игры «сбор уточек» на **Flutter/Dart**. Документ объясняет, как устроен код, как течёт информация и какие правила нельзя нарушать.

## Главный принцип

> **Приложение ничего не считает. Считает backend.**

Клиент только:
1. показывает состояние, которое прислал backend;
2. собирает команды игрока (коды `1..4`);
3. отправляет их на backend и показывает ответ/ошибку.

Никакой симуляции движения, подсчёта очков или проверки правил на клиенте нет и быть не должно. Это и учебное правило, и защита от «рассинхрона» с сервером.

## Слои

```
┌──────────────────────────────────────────────┐
│  UI  (lib/ui/)                                 │  виджеты, экран, отрисовка поля
│  — читает состояние, вызывает методы           │
└───────────────┬────────────────────────────────┘
                │ слушает (Provider / ChangeNotifier)
┌───────────────▼────────────────────────────────┐
│  STATE  (lib/state/game_controller.dart)        │  очередь команд, загрузка,
│  — хранит последнее состояние, очередь хода      │  ошибки, live-обновления
└───────────────┬────────────────────────────────┘
                │ вызывает
┌───────────────▼────────────────────────────────┐
│  REPOSITORY  (lib/data/game_repository.dart)    │  единый вход к данным игры
│  — абстракция; в тестах подменяется фейком       │
└───────────────┬────────────────────────────────┘
                │ использует
┌───────────────▼───────────────┬─────────────────┐
│  GameApi (REST)               │  LiveClient (SSE) │  lib/data/
│  http → /api/round, /turn…    │  /api/live поток  │
└───────────────┬───────────────┴─────────────────┘
                │ HTTP / SSE
┌───────────────▼────────────────────────────────┐
│  BACKEND (Kotlin/Ktor)  — источник правды        │
└──────────────────────────────────────────────────┘
```

Зависимости идут строго вниз: UI → State → Repository → API. Ни один нижний слой не знает о верхнем. Поэтому любой слой можно тестировать или заменить отдельно.

## Как течёт информация

**Чтение состояния (backend → экран):**
1. При старте `GameController.init()` вызывает `refresh()` → `GET /api/round` + `GET /api/events`.
2. Плюс два «двигателя» обновления: периодический опрос (каждые 3 с) и live-поток `/api/live` (SSE). Live-событие означает «что-то изменилось» → контроллер делает `refresh()`.
3. `notifyListeners()` → виджеты перерисовываются.

**Ход игрока (экран → backend):**
1. Игрок жмёт кнопки — `GameController.addCommand(code)` добавляет код в очередь (максимум 5, только если раунд идёт).
2. Кнопка «Отправить ход» → `submitTurn()` → `POST /api/turn/submit` с `{actor, commands}`.
3. Успех → очередь очищается, `refresh()`. Ошибка (`turn_limit_exceeded`, `simulation_error`, …) → показывается баннером, очередь сохраняется.

## Управление состоянием

Используется **Provider + ChangeNotifier** — самый простой «правильный» способ для Flutter:
- один `GameController` на приложение (создаётся в `main.dart`);
- виджеты подписываются через `context.watch<GameController>()`;
- при изменении вызывается `notifyListeners()`.

Почему не Bloc/Riverpod: для MVP и учёбы Provider проще читать и объяснять. Заменить на другой подход — возможная (но не обязательная) задача.

## Тестируемость

- **models_test.dart** — разбор JSON против реального контракта backend.
- **game_controller_test.dart** — логика очереди и отправки хода на фейковом репозитории.
- **widget_test.dart** — рендер экрана и нажатия команд без реального backend.

Всё это возможно, потому что `GameController` зависит от абстракции `GameRepository`, а не от сети. В тестах подставляется `FakeGameRepository` (см. `test/fakes.dart`).

## Что специально оставлено простым (точки роста)

- Один экран — нет навигации и настроек.
- Live-обновление реализовано как «толчок к опросу», а не тонкая обработка каждого события.
- Нет анимаций перемещения и сбора уточек.
- Адрес сервера задаётся при сборке, а не в интерфейсе.
