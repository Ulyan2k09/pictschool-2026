# Трек: Бэкенд

Цель трека — собрать надежную логику игры: состояние миссии, поле, объекты, события, правила, очки, таймеры и обмен данными между ИИ, мобильным приложением и платформой. Бэкенд должен быть центром системы, где видно, что происходит и почему.

## Локальный запуск MVP

Backend реализован как отдельный Kotlin/Gradle-проект на Ktor.

```bash
cd backend
./gradlew run
```

По умолчанию HTTP API поднимается на `http://localhost:8080`.

Swagger UI доступен на `http://localhost:8080/api/docs`.

OpenAPI YAML доступен на `http://localhost:8080/api/docs/openapi.yaml`.

Подробное описание реализованного функционала находится в [FUNCTIONALITY.md](FUNCTIONALITY.md).

Мини-инструкция по подготовке ноутбука для школьников находится в [STUDENT_SETUP.md](STUDENT_SETUP.md).

Конфиг TCP-интеграции с симуляцией:

```bash
SIM_TCP_HOST=127.0.0.1
SIM_TCP_COMMAND_PORT=5055
SIM_TCP_TELEMETRY_PORT=5056
```

Порт `5055` используется для отправки строки команд вида `1 2 3 4`. Порт `5056` зарезервирован под будущую телеметрию от симуляции.

Минимальная проверка:

```bash
nc -lk 5055
```

В отдельном терминале:

```bash
curl -X POST http://localhost:8080/api/round/start \
  -H 'Content-Type: application/json' \
  -d '{"scenarioId":"default"}'

curl -X POST http://localhost:8080/api/turn/submit \
  -H 'Content-Type: application/json' \
  -d '{"actor":"robot","commands":[1,1,3,1,4]}'

curl http://localhost:8080/api/round
curl http://localhost:8080/api/events
curl -N http://localhost:8080/api/live
```

Если на `SIM_TCP_COMMAND_PORT` не запущена симуляция или временная TCP-заглушка `nc`, корректная команда вернет `simulation_error`, а в журнале появится `turn.failed`.

## Что должно быть в MVP

- API или realtime-канал для мобильного приложения.
- Канал обмена с ИИ-агентом.
- Канал обмена с платформой или симуляцией.
- Модель состояния игры: игрок, поле, объекты, маршрут, задания, очки, таймер, статус миссии.
- Обработка событий: команда игрока, перемещение объекта ИИ, блокировка пути, ответ платформы, ошибка, завершение задания.
- Минимальная система логов.

## Что нужно подготовить куратору

- Описать основные сущности: миссия, игрок, поле, объект, платформа, событие, действие ИИ, задание.
- Подготовить схему обмена данными между компонентами.
- Сделать простой способ запуска локально.
- Подготовить тестовые сценарии: нормальный проход, вмешательство ИИ, потеря связи, ошибка платформы.
- Описать, какие правила школьники могут менять без риска сломать весь проект.

## Задачи для участников

- Разобраться, как состояние игры меняется после событий.
- Изменить правила начисления очков или штрафов.
- Добавить новое игровое событие.
- Добавить правило проверки: можно ли двигать объект в выбранную клетку или зону.
- Настроить таймеры, условия победы и проигрыша.
- Сделать систему устойчивее к ошибкам: повтор команды, задержка ответа, неизвестное событие.
- Подготовить данные для отображения в мобильном приложении.

## Возможные механики

- Очки за выполнение задания и штрафы за ошибки.
- Ограниченное число вмешательств ИИ за этап.
- Таймер миссии и таймеры отдельных заданий.
- Состояния платформы: готова, движется, заблокирована, ошибка, восстановление.
- Состояния объектов: свободен, занят, перемещается ИИ, блокирует путь, недоступен.
- События среды: закрытый проход, перемещение объекта, новая цель, сбой датчика, задержка команды.

## Интеграция с другими треками

- С ИИ: бэкенд отправляет состояние и принимает решения агента как события.
- С mobile: бэкенд отдает состояние миссии и принимает команды игрока.
- С computer-systems: бэкенд отправляет команды движения и получает телеметрию или результат симуляции.

## Критерии готовности

- Состояние игры можно восстановить по событиям или логам.
- По логам видно, какой объект передвинул ИИ и как это повлияло на движение.
- Компоненты не зависят друг от друга напрямую, а общаются через бэкенд.
- Ошибки не останавливают всю систему без понятного сообщения.
- Есть несколько готовых сценариев проверки.
- Куратор может быстро показать, где поменять правило игры.

## Риски

- Если правила размазаны по компонентам, школьникам будет трудно понять систему.
- Если нет единого состояния, финальные прогоны будут непредсказуемыми.
- Если бэкенд молча игнорирует ошибки, команды не смогут их диагностировать.
- Слишком сложная архитектура съест время школы.
